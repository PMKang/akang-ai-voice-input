# 自动更新与检查更新方案

状态：macOS 已实现基础检查、下载与重启安装链路；Windows 自 v1.6.1
起支持同一 GitHub Release 下的 Windows x64 包检查、下载、SHA256 校验、
退出替换与自动重启。

## 目标

让用户无需主动访问仓库，也能知道并获取新版本；同时遵守 macOS 的安装权限边界。

## 检查机制

1. 用户进入“关于”页面时，本次运行内自动查询一次 GitHub Releases；应用启动和日常使用不会主动联网检查。
2. 用户也可点击“检查更新”立即重新查询。
3. 将本地 `CFBundleShortVersionString` 与 Release tag 比较。仅当远端版本更高时显示“待更新”。
4. 网络错误会显示明确状态，不影响录音和其他功能。

## 下载与校验

1. 更新包固定来自同一 GitHub Release 中与当前平台匹配的安装包：macOS
   使用 `macos.zip`，Windows 使用 `windows-x64.zip`。
2. 下载在后台进行，不打断当前使用。
3. macOS 解压后校验 App 的 bundle identifier；Windows 下载相邻的
   `.sha256` 文件并校验压缩包，解压后必须找到 `Noboard.exe`。

## 安装策略

1. 默认采用“发现更新 -> 用户确认下载 -> 用户确认重启安装”。
2. 当应用所在目录可写时，启动一个短生命周期的更新助手：主应用退出后替换 App，并重新打开新版本。
3. 当应用位于 `/Applications` 且当前账户无写入权限时，不伪造“静默成功”：定位已下载的新 App，并明确提示用户拖入“应用程序”完成替换。
4. 不下载源码作为普通用户更新路径；源码仅供开发者自行构建。
5. Windows 在独立 PowerShell 进程中等待主程序退出，备份当前目录后覆盖
   新文件并重新启动；目录不可写时明确提示用户先将软件移到个人文件夹。

## 关于页面

“关于”页应包含：

- 当前版本与更新状态。
- 检查更新按钮。
- 可展开的更新日志时间线。
- GitHub 项目入口、开发者说明、隐私说明，以及公众号和视频号入口。

更新日志以 Markdown 维护，并与 GitHub Release Notes 保持一致。第一版可展示：

- `v1.0.1`：优化悬浮窗实时转写、智能降噪状态，并加入关于页更新机制。
- `v1.0.0`：首个公开版本，支持实时语音输入、表达方式、历史记录、词典和本地统计。
