# 未来共建路线图

这是一份公开待办，不是功能承诺。当前项目以 macOS 为主；未来希望把可复用的部分整理清楚，再分别把 Windows 和移动端做成符合各自系统习惯的产品。

## 方向一：先抽出共享核心

可以优先梳理一组不依赖具体操作系统的能力：

- 表达方式的规则格式、导入导出与版本迁移。
- 将内置表达方式从 Swift 源码迁移为带 `id`、版本和默认标记的资源配置；用户本地方案单独保存，升级时可迁移内置方案而不覆盖用户修改。请求透传、无效输入等核心行为边界仍由代码统一兜底。
- 实时模型的统一适配接口、提示词注入和错误处理。
- 输入统计、Token 与费用预估的本地计算口径。
- 可脱敏的音频与文本回归样本，用于验证整理、方言和请求透传效果。

目标是共享业务规则与测试，而不是把 macOS 的系统代码强行搬到别的平台。

## 方向二：Windows 版本原型

Windows 版最值得先验证的是完整输入链路：

1. 全局快捷键和系统托盘入口。
2. 麦克风采集、音量反馈与悬浮窗。
3. 将最终文字写入当前输入控件，失败时可靠复制到剪贴板。
4. 对接现有 Realtime 模型与本地表达方式。

这里需要 Windows 原生能力，例如键盘事件、音频采集和 UI Automation。欢迎熟悉 WinUI、WPF、.NET、Rust 或 Windows 输入自动化的开发者提出可验证方案。

## 方向三：移动端辅助版

移动端第一阶段更适合做“语音记录与表达方式助手”：录音、实时文本、历史记录、规则管理、复制和系统分享。系统级键盘扩展会受到 iOS 与 Android 的权限、审核和后台限制，需要单独调研后再决定范围。

## 方向四：更多模型适配

希望逐步形成可替换的模型适配层，并补齐以下能力：

- Realtime 语音模型的连接、会话事件与增量文本处理。
- 非实时 ASR 与文本整理组合的对比基线。
- 用量、价格与免费额度的可配置估算。
- 模型能力与地区、账户权限差异的清晰提示。

请勿在 Issue、PR、截图或日志中提交 API Key、Workspace ID、真实录音和隐私文本。

## 方向五：macOS 输入体验继续打磨

- `Fn` 与系统输入法切换冲突的稳定拦截方案。
- 复杂应用、自绘编辑器和多窗口场景的可靠文字写入。
- 更好的噪声门限、语音活动检测与实时转写展示。
- 自动更新、签名与发布流程的工程化。

## 如何一起做

涉及系统权限、全局输入、跨平台框架或模型协议时，建议先开 Issue 描述目标、系统版本、复现方式和可行方案，确认边界后再动手。界面、文档、测试和小型独立能力可以直接提交 PR。

提交前请阅读 [参与开发说明](../CONTRIBUTING.md)，并确保：

- 不提交密钥、账户信息、原始音频或私人记录。
- 不引入来源不明或许可不清晰的依赖。
- 新增能力有可重复的验证步骤。
- 文案准确区分“已实现”“实验中”和“规划中”。

有兴趣但暂时没有完整方案也完全欢迎。把问题讲清楚，本身就是对这个项目的贡献。
