# 语音模型与供应商配置设计（草案）

状态：讨论中，暂不实现多供应商接入。

## 待讨论问题

> Noboard 的“语音模型配置”应如何同时支持阿里云百炼与豆包等供应商：在任一时刻只允许激活一个模型，而保留各供应商独立的凭证、地域、端点和能力配置？其中，哪些字段应由应用内置并随版本发布，哪些允许用户配置，哪些必须只保存在 macOS Keychain 中？当用户切换模型时，如何明确告知“表达方式提示词”“个人词典热词”“情感识别”“文件转写”等能力会发生哪些变化？

建议拿这段问题与其他 AI 或技术同事讨论；重点不是“如何列更多模型”，而是单选模型、能力适配与密钥隔离的边界。

## 已记录：阿里云百炼选型

当前产品核心是实时语音输入，模型按能力而不是仅按厂商分组。

| 产品路径 | 推荐模型 | 主要能力 | 与 Noboard 的关系 |
| --- | --- | --- | --- |
| 默认、提示词整理 | `qwen3.5-omni-flash-realtime` | 实时、Prompt 上下文、多语种、情感 | 当前已接入；保留为默认模型 |
| 高质量上下文 | `qwen3.5-omni-plus-realtime` | 实时、Prompt 上下文、多语种、情感 | 后续作为 Omni 高阶档位；可复用表达方式提示词 |
| 方言与稳定术语 | `fun-asr-realtime` | 实时、热词、多语种及方言 | 后续单独接入；个人词典应映射为热词，不复用表达方式提示词 |

不纳入近期用户配置页：

- `qwen3-asr-flash-realtime`：其情感识别能力对当前“直接成文”主流程增益有限；既无 Omni 的 Prompt 上下文，也无 Fun-ASR 的热词/方言优势。先保留在内部路线图。
- 文件转写模型（如 `fun-asr`、`qwen3.5-omni-plus`）：属于“导入录音文件”功能，不能与实时麦克风模型混为同一配置；待文件转写功能立项后再配置。

## 单激活模型原则

用户界面只允许一个 `activeModelID`，即多选一。切换时：

1. 校验所选模型对应供应商的凭证是否已配置。
2. 显示该模型支持/不支持的能力：Prompt 上下文、热词、方言、情感、说话人分离、实时或文件转写。
3. 不兼容能力不静默降级。例如切到 Fun-ASR 时，应明确提示“表达方式不会发送；个人词典将作为热词使用”。
4. 不删除其他供应商已保存的凭证或用户偏好；只是取消激活。

## 配置分层

### 1. 应用内置的模型目录（可随版本升级）

放在项目的 `Resources/` 中，建议未来为 `VoiceModelCatalog.json`。它只保存公开、非敏感数据：

- provider ID、model ID、显示名称、产品路径；
- 实时/文件转写协议；
- 能力矩阵与音频限制；
- 官方文档、费用与控制台链接；
- 默认地域和不可编辑的协议参数。

目录由应用版本发布，不由用户手动填写。这样可避免用户误填模型 ID 或把已下线模型带入生产流程。

### 2. 用户可变的非敏感偏好

保存在本机 `UserDefaults` / 现有持久化存储：

- `activeModelID`（全局唯一、单选）；
- 每个供应商的已选地域、可选自定义端点开关；
- 用户明确配置过的显示偏好。

建议键空间按供应商隔离，例如 `provider.aliyun.region`、`provider.doubao.region`，而不是继续扩展一组通用的 `apiKey` 字段。

### 3. 凭证与敏感配置

仅保存在 macOS Keychain，按供应商和账户范围隔离：

- 阿里：API Key、Workspace ID、地域（地域若不敏感可留在偏好层）；
- 豆包：待官方方案确认后定义所需的 API Key / App ID / Resource ID 等字段；
- 不写入 JSON、UserDefaults、诊断报告、Git 仓库或导出的测试包说明。

## 页面结构建议

1. **语音模型**：卡片目录与单选激活入口；实时和文件转写分两个分区。
2. **供应商凭证**：按供应商展示配置状态、测试连接与官方控制台入口。
3. **能力影响提示**：在模型卡片和切换确认框中说明 Prompt、热词、词典及费用统计行为。

## 豆包待确认项

接入前必须依据官方文档确认，而不是猜测字段：

`doubao-seed-asr-2-0` 是当前豆包侧的优先候选。它同时覆盖实时语音识别与录音文件转写，并主打上下文推理、专有名词/多音字与多语种识别；产品价值高于仅提供情感识别的 `qwen3-asr-flash-realtime`。视觉理解不属于当前桌面语音输入的首期范围。

它不应被直接标注为“支持表达方式提示词”：模型的上下文理解、系统 Prompt、热词或自定义词表在 API 层是否为同一参数仍需实测。接入顺序建议是先打通实时 WebSocket 的最小转写链路，再测试上下文/词典能力，最后评估文件转写。

## 已确认的豆包接入决策（2026-07-21）

首个豆包实现选择 **流式语音识别 2.0 的双向流式 WebSocket**，而不是非实时文件转写或仅单向流式接口。其“边发送音频、边接收中间/最终文本”的模式与 Noboard 的实时录音、即时预览和结束后粘贴流程一致。

- 产品首期语言边界：中文、英文与方言；暂不为其他语言扩大支持范围。
- 连接凭证：App ID、Access Token 与 Secret Key 均为供应商专属秘密，只能存入 macOS Keychain；App ID 虽不等同于密码，仍与同一连接配置一起保存，避免散落在偏好设置或配置文件。
- 实现方式：新增 `DoubaoStreamingASRClient`，单独处理其 WebSocket 鉴权、音频帧、会话事件和增量文本；不得复用阿里 Qwen 的事件协议。
- 能力标记：模型卡显示“上下文与词典能力待验证”，在成功完成官方最小链路与字典/上下文测试前，不启用“表达方式提示词”开关。

1. 实时 ASR 与文件转写分别使用的模型 ID、WebSocket/HTTP 协议和地域。
2. 鉴权所需字段及其 Keychain 命名范围。
3. 是否支持系统提示词、热词、方言、说话人分离、情感识别。
4. 音频编码、采样率、单次时长/文件大小限制。
5. 费用、余额查询与官方充值链接。
